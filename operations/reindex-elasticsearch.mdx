---
title: Reindexing Elasticsearch / OpenSearch
description: How to reindex an Elasticsearch index in Search API after schema changes, including preflight checks, running the migration, and cleanup.
---

After updating an Elasticsearch index's schema — for example by [adding new fields or document types](https://github.com/alphagov/search-api/blob/main/docs/adding-new-fields.md) — you need to reindex the affected index before the new fields and types can be used.

<Warning>
  **Avoid reindexing in production during working hours, except to fix an outage.**

  Reindexing locks the index for writes, so content is not updated in the search index during the process. Reindexing takes around **2 hours** to complete.

  If you need to update stale content during working hours without a full reindex, see [Fix out-of-date search indices](/operations/fix-out-of-date-search-indices) instead.
</Warning>

<Note>
  You do **not** need to reindex when making a change to the `govuk_document_types` gem. Instead, run the `search:update_supertypes` Rake task to update documents in-place. This can be done during working hours.
</Note>

## What the reindexing process does

1. Locks the Elasticsearch index to prevent writes while data is being copied
2. Creates a new index using the schema defined in the currently deployed version of Search API
3. Copies all data from the old index to the new index
4. Compares the old and new data to check for inconsistencies
5. If everything matches, switches the [index alias](https://www.elastic.co/guide/en/elasticsearch/reference/current/indices-aliases.html) to point to the new index

## Step 1: Preflight checks

Ensure you have sufficient free storage to complete the migration. You can check this in:

- The [AWS OpenSearch cluster health dashboard](https://eu-west-1.console.aws.amazon.com/aos/home?region=eu-west-1#opensearch/domains/blue-elasticsearch6-domain?tabId=clusterHealth)
- The [Total free storage space CloudWatch metric](https://eu-west-1.console.aws.amazon.com/cloudwatch/home?region=eu-west-1#metricsV2?graph=~(metrics~(~(~(expression~'FLOOR*28m1*2f1024*29~label~'FreeStorageSpace~id~'e1~region~'eu-west-1~yAxis~'right))~(~'AWS*2fES~'FreeStorageSpace~'DomainName~'blue-elasticsearch6-domain~'ClientId~'172025368201~(id~'m1~region~'eu-west-1~visible~false)))~view~'timeSeries~stacked~false~region~'eu-west-1~title~'Total*20free*20storage*20space*20*28GiB*29~period~60~stat~'Sum~yAxis~(left~(showUnits~true~min~0)~right~(min~0))~legend~(position~'hidden)~start~'-PT12H~end~'P0D))

If free storage is below **250 GB**, check for old indices left over from a previous migration.

### Check for old indices

Sign in as a production poweruser:

```sh
eval $(gds aws govuk-production-developer -e --art 8h)
```

List all search indices, sorted by name:

```sh
k exec deploy/search-api -- sh -c 'curl -s "$ELASTICSEARCH_URI/_cat/indices?v&s=i"'
```

You should see exactly one index each for `detailed`, `government`, `govuk`, `metasearch`, and `page-traffic`. Example healthy output:

```
health status index
green  open   .kibana_1
green  open   .tasks
green  open   detailed-2023-01-01t21-09-52z-0630e696-9d7f-...
green  open   government-2023-01-01t19-46-41z-e44c324c-644...
green  open   govuk-2023-01-01t23-09-56z-2a4ec293-2123-40f...
green  open   licence-finder
green  open   metasearch-2023-01-01t21-41-47z-4a398b23-8bb...
green  open   page-traffic-2023-01-01t04-32-04z-cd2264ce-3...
```

If you see multiple indices starting with the same name, check their timestamps. Anything older than a week is likely a forgotten leftover from a previous migration. Clean them up using the `search:clean` Rake task:

```bash
bundle exec rake search:clean SEARCH_INDEX=alias_of_index_to_clean_up
```

Where `alias_of_index_to_clean_up` is the prefix before the date/time stamp: `govuk`, `government`, `detailed`, `metasearch`, `page-traffic`, or `all` to clean all indices.

## Step 2: Run the reindex

<Tabs>
  <Tab title="Reindex all indices">
    Run the `search:migrate_schema` Rake task on Search API:

    ```bash
    bundle exec rake search:migrate_schema
    ```
  </Tab>
  <Tab title="Reindex a specific index">
    Specify the `SEARCH_INDEX` environment variable:

    ```bash
    bundle exec rake SEARCH_INDEX=government search:migrate_schema
    ```

    Valid values for `SEARCH_INDEX` are: `govuk`, `government`, `detailed`, `metasearch`, `page-traffic`.
  </Tab>
</Tabs>

To monitor progress, check how many documents have been copied to the new index:

```sh
k exec deploy/search-api -- sh -c 'curl "$ELASTICSEARCH_URI/_cat/indices?v"'
```

---

## Step 3: Clean up

Reindexing does **not** delete the old index — this allows you to roll back if there is a serious problem with the new one. Once you are confident the reindex was successful, delete the old (unaliased) index:

```bash
bundle exec rake search:clean SEARCH_INDEX=alias_of_index_to_clean_up
```

<Warning>
  Avoid leaving old indices around. Stale indices consume storage and may prevent new documents from being indexed if storage fills up.
</Warning>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Failed to switch to new index">
    The final step switches Elasticsearch to the newly created index. If content is written to the database while the reindex is running, the task can fail at the end because it detects a difference in the data.

    **Re-running the reindex task usually fixes this.**
  </Accordion>
  <Accordion title="How to stop a reindex in progress">
    1. Find the ID of the reindexing task:

       ```sh
       k exec deploy/search-api -- sh -c 'curl "$ELASTICSEARCH_URI/_tasks?actions=%2Areindex&pretty"'
       ```

    2. Cancel the task:

       ```sh
       k exec deploy/search-api -- sh -c 'curl -XPOST "$ELASTICSEARCH_URI/_tasks/<task_id>/_cancel"'
       ```

    3. Unlock the old index by running the `search:unlock` Rake task:

       ```bash
       bundle exec rake search:unlock SEARCH_INDEX=alias_of_index_to_unlock
       ```
  </Accordion>
  <Accordion title="How to switch back to the old index">
    If you discover a problem after reindexing and need to roll back, run the `search:switch_to_named_index` Rake task:

    ```bash
    bundle exec rake "search:switch_to_named_index[full_index_name] SEARCH_INDEX=index_alias"
    ```

    Where `full_index_name` is the full name of the old index including its date and UUID — for example:

    ```
    govuk-2018-01-29t17:08:21z-31f39bdb-c62b-4607-8081-19ea87fb1498
    ```

    <Warning>
      Switching back to an old index will cause you to **lose any content updates** published while the new index was live. To recover those changes, [re-present them from Publishing API and Whitehall](/operations/fix-out-of-date-search-indices).
    </Warning>
  </Accordion>
</AccordionGroup>
