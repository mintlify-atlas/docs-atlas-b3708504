title: AWS Secrets Manager
description: How GOV.UK uses AWS Secrets Manager to store and access secrets — retrieving credentials, setting up OTP, rotating secrets, and the ExternalSecret Kubernetes integration.
---

GOV.UK uses [AWS Secrets Manager](https://docs.aws.amazon.com/secretsmanager/latest/userguide/) to store and control access to both machine-readable secrets (such as API keys and database credentials used by applications) and shared human-readable credentials (such as shared service accounts).

Secrets Manager replaced the previous `govuk-secrets` shared password store in February 2024.

## Why Secrets Manager

Using Secrets Manager gives GOV.UK:

- **A durable audit trail** — in a security event, you can find out retrospectively all the times any given item was accessed and by whose IAM credentials.
- **Access control via IAM roles** — access is updated automatically for leavers and joiners, eliminating error-prone manual processes.

## Retrieving a credential from Secrets Manager

<Steps>
  <Step title="Log into the production AWS account">
    Assume the developer role:

    ```bash
    gds aws govuk-production-developer -l
    ```
  </Step>

  <Step title="Open Secrets Manager">
    Go to [Secrets Manager](https://eu-west-1.console.aws.amazon.com/secretsmanager/listsecrets?region=eu-west-1) in the AWS console.
  </Step>

  <Step title="Find your secret">
    Search for `2ndline` to find shared credentials, or search by the secret name if you know it.
  </Step>

  <Step title="Retrieve the secret value">
    Choose the credential you need, then under **Secret value**, select **Retrieve secret value**.
  </Step>
</Steps>

You can also retrieve secrets via the AWS CLI:

```bash
aws secretsmanager get-secret-value \
  --secret-id govuk/my-app/api-key \
  --region eu-west-1
```

## Setting up OTP from a 2FA seed

Some shared accounts require two-factor authentication. Their secrets may include a `2fa-seed` value in the format:

```text
otpauth://totp/YOUR_IDENTIFICATION?secret=YOUR_SECRET
```

Generate a one-time password from the seed using `totp-cli`:

```bash
brew install totp-cli
totp-cli instant
# paste the secret value and press Enter
```

## Rotating a credential

1. Retrieve the credential from Secrets Manager.
2. Click **Edit** and update the secret value.

## Adding a secret for a Kubernetes application

Secrets used by GOV.UK applications on Kubernetes follow a specific flow:

```text
AWS Secrets Manager
    ↓ (pulled by External Secrets Operator)
Kubernetes Secret object
    ↓ (referenced by)
Pod environment variable (via valueFrom.secretKeyRef)
```

### Creating the secret in AWS Secrets Manager

<Steps>
  <Step title="Open Secrets Manager and create a new secret">
    Open the [AWS Secrets Manager console](https://eu-west-1.console.aws.amazon.com/secretsmanager/listsecrets?region=eu-west-1). You must create the secret in all three accounts: integration, staging, and production.

    Select **Store a new secret**, then choose **Other type of secret**.
  </Step>

  <Step title="Enter the key and value">
    Enter the secret **Key** (a unique identifier) and **Value** (the actual secret), then select **Next**.
  </Step>

  <Step title="Set the secret name and description">
    Name the secret following the pattern `govuk/<appname>/<secretname>`. Add a description explaining what the secret is used for and where to get the value.
  </Step>

  <Step title="Tag the secret">
    Add a tag with **Key** `added-manually` and **Value** `true`.
  </Step>

  <Step title="Disable automatic rotation and store">
    Ensure **Automatic rotation** is not enabled. Review and click **Store**. Repeat for all three AWS accounts.
  </Step>
</Steps>

### Defining an ExternalSecret in Kubernetes

Create a `.yaml` file in your app's folder under [`charts/external-secrets/templates/`](https://github.com/alphagov/govuk-helm-charts/tree/main/charts/external-secrets/templates/):

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: frontend-elections-api
  labels:
    {{- include "external-secrets.labels" . | nindent 4 }}
  annotations:
    kubernetes.io/description: >
      Credentials used by Frontend to access external Elections API service.
      Field names are "key" for the API key and "url" for the endpoint URL.
spec:
  refreshInterval: {{ .Values.externalSecrets.refreshInterval }}
  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore
  target:
    name: frontend-elections-api
  dataFrom:
    - extract:
        key: govuk/frontend/elections-api
```

After merging, check the Argo CD dashboard to confirm the `external-secrets` chart synced successfully in each environment:

- [Integration](https://argo.eks.integration.govuk.digital/applications/external-secrets)
- [Staging](https://argo.eks.staging.govuk.digital/applications/external-secrets)
- [Production](https://argo.eks.production.govuk.digital/applications/external-secrets)

### Referencing the secret in app Helm values

In `charts/app-config/values-<environment>.yaml`, add a `secretKeyRef` to your app's `extraEnv`:

```yaml
- name: my-app
  helmValues:
    extraEnv:
      - name: ELECTIONS_API_KEY
        valueFrom:
          secretKeyRef:
            name: frontend-elections-api
            key: key
      - name: ELECTIONS_API_URL
        valueFrom:
          secretKeyRef:
            name: frontend-elections-api
            key: url
```

<Warning>
  Do not create new shared credentials or shared user accounts. For any new service, all human users should have individual accounts linked to their identity via single sign-on. If in doubt, ask [govuk-platform-engineering@](mailto:govuk-platform-engineering@digital.cabinet-office.gov.uk).
</Warning>
