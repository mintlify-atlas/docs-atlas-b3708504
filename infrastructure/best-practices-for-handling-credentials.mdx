---
title: Best practices for handling credentials
description: How to safely store, use, and protect credentials on your workstation — AWS, GCP, and general guidance for GOV.UK developers.
---

Technologists occasionally need to access sensitive systems from their workstations — for example, using the AWS CLI to examine an S3 bucket, or an API key to test a third-party API.

Developing software introduces supply-chain risk: malware on a workstation could steal credentials used to access sensitive systems. The following practices reduce this risk.

## AWS credentials

<CardGroup cols={2}>
  <Card title="MUST: Use gds-cli / aws-vault" icon="shield-check">
    Never store credentials in `~/.aws/credentials`. Use `gds-cli` or `aws-vault` instead — these store credentials in the macOS keychain.
  </Card>
  <Card title="SHOULD: Use least-privilege roles" icon="user-lock">
    Use the role with the lowest privileges that will allow you to complete your task. If you only need read access, use a `readonly` role.
  </Card>
</CardGroup>

### Use gds-cli to assume roles safely

```bash
# Assume a role and export credentials to the current shell
eval $(gds aws govuk-production-developer -e --art 8h)
export AWS_REGION=eu-west-1
```

<Warning>
  Avoid running package manager commands (such as `npm install`) in shell sessions with AWS environment variables set. If you open a shell with `gds aws govuk-production-developer -s`, use that shell only for AWS interactions. Untrustworthy package scripts could read your environment and exfiltrate credentials.
</Warning>

## GCP credentials

Revoke local GCP auth credentials as soon as you have finished working:

```bash
gcloud auth revoke
gcloud auth application-default revoke
```

Leaving GCP credentials active on your workstation unnecessarily increases the window of exposure if malware runs.

## Other credentials

When storing any credentials in plain-text files (for example, `.env` files):

- Ensure the credentials have the **minimum set of access required** for the task.
- **Delete them when you no longer need them.** Even if the file is only local, it could be stolen by malware on your workstation.

<Warning>
  Never commit credentials to Git. GOV.UK repositories use tools such as Brakeman, CodeQL, and Dependency Review in CI to detect secrets — but prevention is better than detection.
</Warning>

## Git and commit signing

Pushes to the `main` branch are prevented by GitHub configuration. However, there is still a risk that malware could automatically commit code without your knowledge.

Consider updating your workflow to require an interactive step before pushing commits to GitHub. You can do this by:

- Configuring Git to sign commits using a method that requires an interactive step, such as touching a YubiKey.
- Disabling your SSH agent so push/pull commands require your SSH password.

See the [YubiKey setup guide](https://docs.publishing.service.gov.uk/manual/setup-a-yubikey.html) for instructions on using a hardware key for commit signing.

## Summary of requirements

| Requirement | Severity |
|---|---|
| Do not store AWS credentials in `~/.aws/credentials` | MUST |
| Use gds-cli or aws-vault to manage AWS credentials | MUST |
| Do not run package manager commands in shells with AWS credentials | SHOULD |
| Use the least-privilege role for each task | SHOULD |
| Revoke GCP credentials when finished | SHOULD |
| Avoid storing credentials in plain-text files; delete when no longer needed | SHOULD |
| Sign Git commits with an interactive step (e.g. YubiKey) | COULD consider |
